## Install (recommended) — via esp32_balboa_spa

1. Clone firmware (includes this UI as submodule):
   ```bash
   git clone --recursive https://github.com/shomanjk/esp32_balboa_spa.git
   ```
   Or clone then: `git submodule update --init --recursive`
2. Copy `src/config-example.h` → `src/config.h`; set Wi‑Fi, MQTT, RS485 pins — see [M5 Atom Lite + Atomic RS485 Base](https://github.com/shomanjk/esp32_balboa_spa#m5-atom-lite--atomic-rs485-base-tub-side).
3. Build and flash (example tub env):
   ```bash
   pio run -e M5AtomLite-tub -t upload
   pio run -e M5AtomLite-tub -t uploadfs
   ```
4. On your LAN, open the gateway (e.g. `http://spa-<mac>.local/`). Login accepts any username/password on the local gateway (no Balboa cloud account).

Full build flags, OTA, MQTT/HA: [esp32_balboa_spa README](https://github.com/shomanjk/esp32_balboa_spa).

Also see: [Build with PlatformIO](https://github.com/shomanjk/esp32_balboa_spa#build-with-platformio) · [Web interface](https://github.com/shomanjk/esp32_balboa_spa#web-interface) · [MQTT and Home Assistant](https://github.com/shomanjk/esp32_balboa_spa#mqtt-and-home-assistant)

**Releases:** [github.com/shomanjk/esp32_balboa_spa/releases](https://github.com/shomanjk/esp32_balboa_spa/releases)

## Two UIs on the device

When you run [esp32_balboa_spa](https://github.com/shomanjk/esp32_balboa_spa), the gateway serves two related web experiences:

- **Firmware portal (fully wired for commands):** `/status`, `/config`, `/state` — RS485 commands via `/devices/sci` and `/api/status/controls`.
- **LittleFS `balboa-spa` tab (this PWA):** bundled from `dist/`; treat controls as **read-first** unless verified on your build (same caveat as [esp32_balboa_spa — Web interface](https://github.com/shomanjk/esp32_balboa_spa#web-interface)).

## Local backend (this fork)

This UI is built and shipped inside **[shomanjk/esp32_balboa_spa](https://github.com/shomanjk/esp32_balboa_spa)**. The ESP32 talks to the spa over **RS485** (no Balboa Cloud Connect required). Optional: MQTT, Home Assistant discovery, TCP bridge (port 4257).

Upstream SPA origin: [jozefnad/balboa-spa](https://github.com/jozefnad/balboa-spa) · Protocol reference: [ccutrer/balboa_worldwide_app `doc/protocol.md`](https://github.com/ccutrer/balboa_worldwide_app/blob/master/doc/protocol.md).

## Features

Users can control:

- Target temperature
- Time
- Heat mode
- Hold mode
- Ranges
- Pumps
- Blowers
- Auxs
- Lights
- Filter cycles
- etc.

On the local gateway, command coverage depends on firmware and route (`/status` vs LittleFS tab); see [esp32_balboa_spa](https://github.com/shomanjk/esp32_balboa_spa).

## Legacy: Balboa Cloud PWA

The original [jozefnad/balboa-spa](https://github.com/jozefnad/balboa-spa) targeted Balboa’s cloud API:

- Requires a Balboa WiFi module, Spa Control app, and **Cloud Connect**.
- API: Balboa Cloud (`bwgapi`).
- Public demo (cloud): [jozefnad.github.io/balboa-spa](https://jozefnad.github.io/balboa-spa/) — **not** the same as the UI served by esp32_balboa_spa on your LAN.

This fork’s **primary** deployment path is the local gateway above.

For local tub builds, the device is the API host — you do not need cloud `.env` settings from the template `env` file (see comments in [`env`](env)).

## Fork Workflow Policy

For this fork, treat `src/` as the source of truth.

- Default commits should include source files (for example `src/**`) and docs.
- Build artifacts in `dist/` are generated by Vite and can be rebuilt locally.
- Local environment files (`.env`, `dist/.env`) should not be committed.
- End users should not need to run `npm run build` unless developing the SPA.

### Build and deploy workflow

1. Edit source in `src/`.
2. Build assets:
   - `npm run build`
3. Deploy filesystem assets from `dist/` via **[shomanjk/esp32_balboa_spa](https://github.com/shomanjk/esp32_balboa_spa)** (`balboa-spa` git submodule; PlatformIO `data_dir = balboa-spa/dist`; `pio run -e <env> -t uploadfs`).
4. After SPA changes, bump the `balboa-spa` submodule commit in the firmware repo so releases pick up the new `dist/`.
